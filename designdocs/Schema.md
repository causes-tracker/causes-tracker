# Data Model

This document defines the entity model for Causes.
The proto service definitions (see [ADR-009](Decisions.md#adr-009-api-specification-format--protobuf)) are the authoritative contract; this document describes the conceptual model they encode.
Field names here are conceptual; generated code may differ.

## Overview

The core entities are organised around **Projects**, which contain **Signs**, **Symptoms**, and **Plans**.
**Plans** link to Signs and Symptoms to express intent: "this change addresses these observed problems".
**Comments** carry discussion on Symptoms and Plans.
Every top-level resource participates in a per-resource **Journal** and may carry a **FederationProvenance** record if it arrived from another instance.

```
Instance
└── Project
    ├── Sign          (machine-observed, typically sourced from external systems)
    ├── Symptom       (human-reported)
    │   └── Comment
    └── Plan          (proposed change)
        ├── → Sign     (many-to-many)
        ├── → Symptom  (many-to-many)
        ├── → Plan     (depends-on, many-to-many)
        └── Comment
```

## Common fields

Every Sign, Symptom, Plan, and Comment carries these fields.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Assigned by the origin instance at creation. |
| `project_id` | UUID | Owning project. |
| `created_at` | timestamp | Set at creation; immutable. |
| `updated_at` | timestamp | Updated on each journal entry. |
| `embargoed` | bool | If true: not federated, not publicly visible, excluded from search and feeds. |
| `provenance` | FederationProvenance? | Absent for locally-authored resources. |

## FederationProvenance

Carried by every resource received from another instance.
See [Federation trust and remote identity](Decisions.md#adr-010-security-model) for the full model.

| Field | Type | Notes |
|---|---|---|
| `origin_instance_id` | UUID | Instance that created this resource. |
| `origin_id` | UUID | Resource's `id` on the origin instance. |
| `received_from` | UUID | `instance_id` of the direct upstream that delivered this copy. |
| `relay_hops` | UUID[] | `instance_id`s of instances between origin and `received_from`, in order. Empty if `received_from == origin_instance_id`. |

`(origin_instance_id, origin_id)` is the globally stable identity of a federated resource.
Locally-authored resources have no provenance record; `id` alone is their global identity.

## ResourceJournal

Every Sign, Symptom, Plan, and Comment has an append-only journal of versions.
The journal is the unit of replication: a downstream requests entries with `version > N` to catch up incrementally.
Deletes are logical — a `delete` entry is appended; the resource is not physically removed.
Edits are authored on the origin instance only; downstream instances are read replicas.

| Field | Type | Notes |
|---|---|---|
| `version` | uint64 | Monotone per-resource sequence number, assigned by the origin instance. |
| `at` | timestamp | |
| `by` | FederatedIdentity | `(instance_id, local_user_id)` of the author. |
| `op` | enum | `create`, `update`, `delete`, `undelete` |
| `snapshot` | bytes | Full serialised resource state at this version. |
| `provenance` | FederationProvenance? | How this entry arrived; absent for locally-authored entries. |

## Project

The organisational unit.
All Signs, Symptoms, and Plans belong to exactly one Project.
Projects belong to one Instance and are not themselves federated — federation operates at the resource level.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `name` | string | Human-readable; unique within an instance. |
| `description` | string | Markdown. |
| `visibility` | enum | `public` (readable by anonymous users) or `private` (authenticated users only). |
| `embargoed_by_default` | bool | New Signs, Symptoms, and Plans start embargoed when true. |
| `created_at` | timestamp | |

## Sign

Machine-observed evidence of a problem.
Signs are typically generated by external systems (crash reporters, test suites, monitoring) and pulled in by connectors.
Causes stores a summary or reference; the source system holds the full data.
Signs are usually worked with in aggregate rather than individually.

| Field | Type | Notes |
|---|---|---|
| *(common fields)* | | |
| `title` | string | Short human-readable description. |
| `source` | string | System of origin (e.g. `sentry`, `datadog`, `test-suite`). |
| `source_ref` | string? | URL or identifier in the source system. |
| `observed_at` | timestamp | When the sign was first observed. |
| `last_observed_at` | timestamp | Most recent occurrence. |
| `count` | uint64 | Number of occurrences as reported by the source. |
| `affected_version` | string? | Project version string where this was observed. |
| `severity` | enum | `info`, `warning`, `error`, `critical` |

Signs do not have Comments; discussion belongs on the Symptoms and Plans they inform.

## Symptom

A human-reported experience of a problem.
Equivalent to a bug report in traditional trackers.
Carries discussion via Comments and workaround information.

| Field | Type | Notes |
|---|---|---|
| *(common fields)* | | |
| `title` | string | Short description. |
| `description` | string | Markdown; the full report including reproduction steps and workarounds. |
| `reporter` | FederatedIdentity | Who filed it. |
| `status` | enum | `open`, `in-progress`, `resolved`, `wontfix`, `duplicate` |
| `duplicate_of` | UUID? | Canonical Symptom when `status == duplicate`. |
| `affected_versions` | string[] | Versions the reporter has observed this in. |

## Plan

A developer's proposed change, linked to the Signs and Symptoms it addresses.
May stand alone (speculative work, pre-emptive fixes) or be anchored to one or more Signs/Symptoms.
Developer diagnosis and investigation discussion occurs on the Plan.

| Field | Type | Notes |
|---|---|---|
| *(common fields)* | | |
| `title` | string | Short description. |
| `description` | string | Markdown; includes diagnosis, design notes, and approach. |
| `author` | FederatedIdentity | Who proposed it. |
| `status` | enum | `proposed`, `accepted`, `in-progress`, `done`, `abandoned` |
| `linked_signs` | UUID[] | Signs this plan addresses. |
| `linked_symptoms` | UUID[] | Symptoms this plan addresses. |
| `depends_on` | UUID[] | Plans that must complete before this one can proceed. |
| `target_version` | string? | Intended release. |

## Comment

Discussion on a Symptom or Plan.
Comments are first-class resources with their own Journal entries and FederationProvenance.
This means a comment thread can be federated independently: when a company publishes a Plan to a public instance, community replies are new Comment resources originating on the public instance and replicated back.

| Field | Type | Notes |
|---|---|---|
| *(common fields)* | | |
| `parent_id` | UUID | The Symptom or Plan this comment belongs to. |
| `parent_type` | enum | `symptom`, `plan` |
| `author` | FederatedIdentity | |
| `body` | string | Markdown. |

_Open: flat vs threaded comment model._

## User

A person with an account on this instance.
Email addresses are stored but never exposed in general API responses (see [ADR-010](Decisions.md#adr-010-security-model)).

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `display_name` | string | Shown in UI and shared with federation peers (no other personal data is federated). |
| `email` | string | Stored for authentication; visible only in explicit admin views. |
| `auth_provider` | string | Identifier of the IdP used at registration. |
| `created_at` | timestamp | |

Role assignments are stored separately as `(user_id, project_id?, role)` tuples.
`project_id` is absent for instance-level roles.
See [Authorisation model](Decisions.md#adr-010-security-model) for the proposed role set.

## ServiceAccount

Represents a non-human principal: the BFF, a connector, or an inbound federation peer.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `display_name` | string | e.g. `bff`, `github-connector`, `causes.example.org` |
| `kind` | enum | `bff`, `connector`, `federation` |
| `remote_instance_id` | UUID? | Set when `kind == federation`; the `instance_id` of the remote. |
| `created_at` | timestamp | |
| `last_used_at` | timestamp? | |

Service session tokens are issued against a ServiceAccount and inherit its role assignments.

## FederationPeer

A remote instance that this instance has a trust relationship with.
The `inbound_account_id` is the ServiceAccount record for the remote instance's inbound connection; permissions on that account govern what the remote may do here.

| Field | Type | Notes |
|---|---|---|
| `instance_id` | UUID | Stable identity of the remote instance. |
| `contact_url` | string | Current reachable URL. |
| `outbound_account_id` | UUID | ServiceAccount used when this instance connects to the remote. |
| `inbound_account_id` | UUID | ServiceAccount representing the remote's inbound connection here. |
| `established_at` | timestamp | |

In the UI, `contact_url` is displayed as hostname only (e.g. `causes.example.org`).

## Open questions

- **Release / version model:** The project timeline (releases, milestones, associated commits) is mentioned in the Manifesto but not yet modelled here.
  Signs and Plans reference version strings directly for now; a first-class Release entity will be needed.
- **Tags / labels:** No label model is defined yet.
  Signs and Symptoms likely benefit from structured tags for filtering and automation.
- **Comment threading:** Flat model assumed; threaded replies are not yet designed.
- **Sign aggregation:** The relationship between individual Sign observations and aggregated Sign groups (e.g. a Sentry issue containing many events) is not yet modelled.
