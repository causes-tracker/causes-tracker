causes/v1/annotation.proto¶

Annotation carries structured analysis data attached to a Sign.

Annotation¶

Annotation carries structured or machine-generated analysis data attached to a Sign. Annotations differ from Comments in that:

  • They are typically machine-authored (by a connector or analysis service).
  • Their content may be structured binary data rather than Markdown.
  • The annotation_type field names the schema so consumers can decode content.

Example: a crash-report Sign may have an Annotation of type "backtrace/v1" carrying the retraced backtrace and variable values produced by a retrace service. Storing this as an Annotation rather than a Comment keeps structured analysis artifacts separate from human narrative discussion.

Human narrative discussion of a Sign belongs in Comment messages with parent_type == PARENT_TYPE_SIGN.

annotation_type follows the convention "domain/version" (e.g. "backtrace/v1", "heap-summary/v1", "log-context/v1"). Well-known types are documented in the connector specification. Connectors and analysis services may define their own types.

Field Type Label Description
meta ResourceMeta
parent_id SignId parent_id is the Sign this annotation belongs to.
annotation_type string annotation_type identifies the content schema. Convention: "domain/version" (e.g. "backtrace/v1").
source_type Annotation.SourceType
author FederatedIdentity
content bytes content is the annotation payload; its schema is defined by annotation_type.

Annotation.SourceType¶

Name Number Description
SOURCE_TYPE_UNSPECIFIED 0
SOURCE_TYPE_CONNECTOR 1 SOURCE_TYPE_CONNECTOR: generated by a data-source connector or analysis service.
SOURCE_TYPE_HUMAN 2 SOURCE_TYPE_HUMAN: manually authored by a person.

AuthService¶

AuthService handles human authentication via the server-driven Google Device Authorization Grant (RFC 8628). The client never holds the OAuth client secret — the server drives the entire flow.

Login is two RPCs (not streaming) so that pending state survives server restarts — the database holds device-flow state between polls.

Method Name Request Type Response Type Description
StartLogin StartLoginRequest StartLoginResponse StartLogin initiates a device-flow login. The server contacts the IdP and returns a user code and verification URL for the human to complete in a browser, plus a nonce for polling.
CompleteLogin CompleteLoginRequest CompleteLoginResponse CompleteLogin polls for login completion. The client calls this repeatedly with the nonce from StartLogin until the server returns a session token or an error.
WhoAmI WhoAmIRequest WhoAmIResponse WhoAmI returns the identity of the currently authenticated caller.

causes/v1/comment.proto¶

Comment carries human discussion on a Sign, Symptom, or Plan.

Comment¶

Comment carries human discussion. Comments are first-class resources with their own journal and ResourceOrigin, so community comment threads on a published resource can federate back to the origin instance independently.

Comments may be attached to Signs as well as Symptoms and Plans. For machine-generated structured data on a Sign (e.g. a retraced backtrace), prefer Annotation (see annotation.proto) over Comment.

ReplicationPath¶

ReplicationPath records the transit path for a specific journal entry. This is per-entry information (it can differ if the same entry arrives via multiple relay paths); it belongs on JournalEntry, not on the resource snapshot. Separating ReplicationPath from ResourceOrigin avoids the ambiguity that would arise if both the resource snapshot and its enclosing JournalEntry carried a combined "provenance" field with overlapping semantics.

Field Type Label Description
received_from InstanceId received_from is the instance_id of the direct upstream that delivered this entry.
relay_hops InstanceId repeated relay_hops lists instance_ids between the origin and received_from, in order. Empty when received_from == origin instance_id.

ResourceMeta¶

ResourceMeta contains fields present on every top-level resource. Embed as field 1 of every Sign, Symptom, Plan, Comment, Annotation, and link type rather than repeating these definitions.

project_id is the local project on this instance under which the resource is filed. It is not part of the stable resource identity (see ResourceOrigin). Project mapping is the responsibility of the connection originator: when pushing, the originator translates its project_id to the destination's before serialising; when pulling, it translates the source's project_id to its own before storing. The snapshot therefore always arrives with the correct local project_id; no rewriting on ingest is required. See ADR-012.

Field Type Label Description
id string
project_id ProjectId
created_at google.protobuf.Timestamp
updated_at google.protobuf.Timestamp
embargoed bool embargoed suppresses federation and public visibility (see ADR-010).
origin ResourceOrigin origin is absent for resources created on this instance.

ResourceOrigin¶

ResourceOrigin identifies where a resource was originally created. Absent (message not set) for resources created on this instance. Once set at creation, origin is immutable across all journal entries. (instance_id, id) is the globally stable identity of a federated resource, independent of relay path; used for deduplication when the same resource arrives via multiple inbound paths.

Field Type Label Description
instance_id InstanceId instance_id is the Causes instance that created this resource.
id string id is the resource's local identifier on the origin instance. Stored as a plain string because at this level the resource type — and therefore the appropriate typed ID message — is not known.

causes/v1/link.proto¶

Relationship links between top-level resources.

Links are stored as independent intermediate objects rather than as repeated ID fields on the parent resource for two reasons:

  1. A downstream instance can create a local link between resources without requiring write access to either linked resource. For example, a downstream can link a local Sign to a Plan without sending a change to wherever that Plan originates. Whether downstream instances can also propose edits to the Plan itself is an open design question (see ADR-010 provisional note on cross-instance edit proposals); the link object is correct regardless of how that question resolves.

  2. Embargo propagation: if either linked resource is embargoed, the link itself must be embargoed (the server enforces this invariant). Storing links separately makes embargo checks tractable — the link can be hidden without revealing which embargoed resource it referenced.

causes/v1/plan.proto¶

Plan represents a developer's proposed change.

Plan¶

Plan is a developer's proposed change, linked to Signs and Symptoms it addresses. Developer diagnosis and investigation discussion occurs on the Plan via Comments.

Relationships to Signs, Symptoms, and other Plans are stored as independent intermediate objects (PlanSignLink, PlanSymptomLink, PlanDependency in link.proto) rather than as repeated ID fields here. This allows downstream instances to create local links to federated Plans without modifying the federated resource.

causes/v1/sign.proto¶

Sign represents a machine-observed piece of evidence of a problem.

Sign¶

Sign is machine-observed evidence of a problem. Signs are typically sourced from external systems (crash reporters, test suites, monitoring) via connectors. Causes stores a summary or reference; the source system holds the full data. Signs are usually worked with in aggregate.

Structured analysis data (e.g. a retraced backtrace from a crash dump, a heap summary, log context) is attached via Annotation messages (see annotation.proto). Human narrative discussion — once a developer or user has an opinion — belongs on linked Symptoms and Plans. This separation is a design choice, not a hard rule: human Comments on Signs are also permitted (see comment.proto) for cases where brief annotation is more appropriate than creating a full Symptom.

Field Type Label Description

title string
source string source identifies the system of origin (e.g. "sentry", "datadog", "test-suite").
source_ref string source_ref is the URL or identifier of this Sign in the source system.
observed_at google.protobuf.Timestamp
last_observed_at google.protobuf.Timestamp
count uint64 count is the number of occurrences as reported by the source system.
affected_version string affected_version is the project version string where this was observed.
severity Sign.Severity
